# Wyshbone UI Diagnostic Report — What the UI Actually Does

_Generated from codebase analysis. This is not aspirational; it describes current behaviour._

---

## 1. What You Send to Supervisor

### Payload Structure

When a user submits a search, the server constructs a `requestData` object and inserts it into the `supervisor_tasks` table in Supabase. The Supervisor picks it up asynchronously.

**Actual TypeScript interface** (`server/supabase-client.ts`):
```typescript
interface SupervisorTaskData {
  user_message: string;
  search_query?: {
    business_type?: string;
    location?: string;
    requested_count?: number;
  };
  demo?: string;
  metadata?: Record<string, any>;
}
```

**What actually gets sent varies by dispatch path** (from `server/routes.ts`):

**Direct dispatch (RUN_SUPERVISOR):**
```json
{
  "user_message": "find 5 pubs in York with live music",
  "original_user_message": "find 5 pubs in York with live music",
  "search_query": {
    "business_type": "pubs",
    "location": "york",
    "requested_count": 5
  }
}
```
Plus optionally: `google_query_mode`, `metadata`, `scenario`, `constraints` (if provided by the client). Does NOT include `search_query.attributes` — see weakness #1.

**Clarify resolution path:** Same shape but `search_query` may include fields from the clarify session.

**Followup reuse path:** Same shape, plus `search_query.attributes` if the prior saved intent had them.

**CLARIFY_FOR_RUN delegated to Supervisor:** Adds `supervisor_preflight` with the routing reason.

### How Intent Is Captured

1. **`decideChatMode()`** (`server/lib/decideChatMode.ts`) classifies the message into `CHAT_INFO`, `CLARIFY_FOR_RUN`, or `RUN_SUPERVISOR` using regex-based entity/location detection against a hardcoded list of ~200+ UK/IE cities and counties.

2. **`detectSupervisorIntent()`** (`server/intent-detector.ts`) extracts `business_type`, `location`, and `requested_count` using regex. The `taskType` defaults to `find_prospects`.

3. **`sanitizeBusinessType()`** strips quantifiers ("find **5** pubs" → `requested_count: 5`, `business_type: "pubs"`).

### How Attributes Are Expressed

**`extractAttributeConstraints()`** (`server/lib/decideChatMode.ts`) uses regex to pull "with X" and "and have/serve/are X" phrases from the user message. Returns `string[]`.

**Critical gap: attributes are NOT added to `search_query` on the direct `RUN_SUPERVISOR` path.** They are extracted (line 2040) but only used to populate `allConstraintDescriptors` for clarification gating — if all constraints pass the gate, the attributes are never injected into `search_query.attributes`.

Attributes DO reach `search_query.attributes` in exactly two cases:
- **Followup reuse path** (line 1687): `requestData.search_query.attributes = lastIntent.attributes` — but only if a prior intent saved them.
- **Clarify resolution path** (line 1618): saved to `lastIntent` via `currentSession.semantic_constraint`.

**On the primary direct-dispatch path** (line 1776), `attributeConstraints` is extracted but never assigned to `requestData.search_query.attributes`. The Supervisor receives `search_query` without attributes unless the user went through clarification first.

## 2. How You Represent Constraints

### What You Send

Constraints are represented as **free text and string arrays**, not typed structured objects. There is no constraint schema, no hard/soft classification, no evidence requirements, and no constraint metadata in the dispatch payload.

| Field | Type | Contents |
|-------|------|----------|
| `search_query.business_type` | `string` | Entity noun, regex-extracted |
| `search_query.location` | `string` | City/county, validated against hardcoded list |
| `search_query.requested_count` | `number` | Integer count from user text or clarification |
| `search_query.attributes` | `string[]` | **Only present on followup/clarify paths** |
| `metadata.constraints` | `Record<string, any>` | Opaque bag from the client (scenario knobs) |

### What You Don't Send

- No `hardness` flag per constraint (hard vs soft)
- No `evidence_required` metadata
- No structured constraint objects with `field`, `operator`, `value`
- No constraint IDs for the Supervisor to reference in responses

The constraint typing, hardness, and structure you see in the UI (`constraintsExtracted` artefact with `kind`, `hardness`, `field`, `op`, `value`) is entirely **generated by the Supervisor/Tower** on the backend — the UI never sends that structure, it only receives and renders it.

## 3. What You Receive Back

The UI polls `supervisor_task_artefacts` for rows matching the `run_id`. Each artefact has a `type` string and a JSON `payload`. The following types are parsed in `chat.tsx` `parseSiblingArtefacts()`:

### `delivery_summary` (Primary result, always present)
```typescript
{
  status: "PASS" | "PARTIAL" | "STOP" | "FAIL" | "ACCEPT" | "COMPLETED",
  stop_reason?: { message: string, code?: string } | string,
  delivered_exact: DeliveryLead[],      // Leads meeting all criteria
  delivered_closest: DeliveryLead[],    // Near-miss or fallback leads
  shortfall: number,
  requested_count: number,
  delivered_count: number,
  verified_exact_count: number,
  suggested_next_question?: string,
  relaxations?: string[]
}
```

### `constraints_extracted` (When Supervisor parses constraints)
```typescript
{
  requested_count_user: number,
  constraints: Array<{
    id: string,
    kind: "location" | "attribute" | "category",
    hardness: "hard" | "soft",
    field: string,
    op: string,
    value: any,
    label: string
  }>
}
```

### `lead_verification` (Per-lead verification, one artefact per lead)
```typescript
{
  lead_id: string,
  lead_name: string,
  lead_index: number,
  verified_exact: boolean,
  all_hard_satisfied: boolean,
  location_confidence: string,
  constraint_checks: Array<{
    constraint_id: string,
    status: "yes" | "no" | "search_bounded" | "unknown",
    hard: boolean,
    reason: string,
    evidence_id?: string
  }>
}
```

### `tower_semantic_judgement` (LLM-based evaluation)
```typescript
{
  lead_id: string,
  lead_name: string,
  tower_status: "strong_match" | "match" | "weak_match" | "no_match" | "rejected",
  confidence: number,
  tower_matched_snippets: string[],
  constraint_to_check: string
}
```

### `verification_summary` (Global verification stats)
```typescript
{
  verified_exact_count: number,
  requested_count_user: number,
  constraint_results: Array<{ constraint_id: string, status: string }>,
  budget_used: number,
  budget_total: number
}
```

### `run_receipt` (Final tally)
```typescript
{
  unique_email_count: number,
  unique_phone_count: number,
  contacts_proven: boolean,
  requested_count: number,
  delivered_count: number,
  narrative_lines: string[],
  tower_verdict: string,
  outcomes: {
    attributes: Array<{
      attribute_raw: string,
      matched_count: number,
      evidence_refs: string[]
    }>
  }
}
```

### What May Be Missing

The verification artefacts (`lead_verification`, `tower_semantic_judgement`, `verification_summary`) are produced only when the Supervisor's verification layer runs — which depends on budget, constraint type, and backend configuration. The UI has no way to force or request verification. When these artefacts are absent, the UI falls back to the `delivery_summary` alone, meaning all verification-dependent rendering (badges, evidence, per-constraint status) is effectively dormant.

## 4. How You Decide to Show "Verified"

### The Three Verification Sources

`RunResultBubble` checks three sources to determine `hasVerificationData`:

```typescript
const hasReceiptAttrData = receiptAttrsForIds.size > 0;
const hasSemanticData = semanticJudgements.length > 0;
const hasVerificationData = leadVerifications.length > 0
  || hasSemanticData
  || hasReceiptAttrData;
```

### Building the Verified Set

When `hasVerificationData` is true:
1. **`leadVerifications`**: A lead is "verified" if `verified_exact === true` OR `all_hard_satisfied === true` OR all hard `constraint_checks` have `status === 'yes'`.
2. **`semanticJudgements`**: A lead is "verified" if `tower_status` is `'strong_match'` or `'match'`.
3. **`receiptAttrsForIds`**: Leads whose IDs appear in `runReceipt.outcomes.attributes[].evidence_refs`.

Verified lead IDs go into a `Set<string>`. Everything else is a candidate.

### Badge Assignment (`getLeadBadgeStatus`)

```
If lead ID is in verifiedIds → 'verified' (green checkmark)
If lead ID is in weakMatchIds → 'weak_match' (blue badge)
Otherwise → defaultBadgeStatus:
  If hasUnverifiedLeads → 'unverified' (grey "Unverified")
  Else → 'candidate' (grey "Candidate")
```

### When No Verification Data Exists (the common case)

When `hasVerificationData` is false (i.e., no `lead_verification`, no `tower_semantic_judgement`, no receipt attribute data):
- `splitLeadsByVerification` returns ALL leads as `matches` (not candidates)
- `hasUnverifiedLeads` is false
- `defaultBadgeStatus` is `'none'`
- **No badges are shown at all** — leads appear neutrally as "Results (N)"

This is the fix applied to prevent the previous bug where all leads showed "Unverified" when verification was never attempted.

## 5. Evidence Display

### What Is Shown

When evidence data exists on a lead object, `LeadRow` renders:

1. **`match_reason`** (or `match_summary`): Shown in green text below the lead name.
2. **`evidence[0].snippet`** (or `.quote` or `.summary`): First evidence item, italicized, truncated to 120 characters.
3. **`evidence_source_url`**: Blue link labelled "Source: [domain]", truncated to 60 characters.

### What Is Not Shown

- **Only the first evidence item is rendered.** If a lead has 5 evidence items, the user sees 1.
- **Evidence is only shown if the lead object has these fields populated.** The UI does not fetch evidence separately — it relies on the backend putting `evidence`, `match_reason`, and `evidence_source_url` directly on the lead objects inside `delivery_summary`.
- **Lead objects in `delivery_summary` may not populate these fields.** The `delivered_exact[]` / `delivered_closest[]` leads contain `name`, `address`, `phone`, `email`, `website` — but `evidence`, `match_reason`, and `evidence_source_url` are only present when the backend enriches them. Evidence also lives in the separate `lead_verification` artefacts, but the UI does not cross-reference lead verification evidence back into individual lead rows.
- **The `VerificationEvidenceView` in `CvlArtefactViews.tsx` CAN render structured evidence** (snippet + source_ref + source_type per constraint), but this component is rendered in the Activity Panel's artefact expansion, not in the main chat result bubble. Most users never see it.
- **No per-constraint evidence breakdown per lead in the result bubble.** The user cannot see "Location: verified via Google Maps" or "Live music: found on their website" in the main results view.

### Honest Assessment

Evidence display rendering code exists and works correctly when the data is present. But the data it needs (`evidence`, `match_reason`, `evidence_source_url` on lead objects) is only populated when the backend's verification layer enriches leads. When verification doesn't run, or runs but stores evidence in separate `lead_verification` artefacts rather than on the leads themselves, the user sees results with no evidence at all. The UI does not bridge this gap.

## 6. Constraint Visibility

### What the User Can See

When a `constraints_extracted` artefact is present:
- A "Constraints" section lists each constraint's `label` (e.g., "Location: York") with a **HardnessBadge** (red for "hard", amber for "soft").
- If `verification_summary` is also present, each constraint gets a **StatusBadge**: green "Verified" (`yes`), red "Not met" (`no`), or grey "Not verified" (`unknown`).

### What the User Cannot See

- **Whether each constraint passed or failed PER LEAD.** The `constraint_checks` array in `lead_verification` artefacts contains per-lead, per-constraint pass/fail with reasons — but this is only rendered in the `TechnicalDetails` expandable section (dev-oriented) and the Activity Panel artefact view. The main result bubble does not show per-lead constraint outcomes.
- **Constraint provenance.** The user cannot see that "live music" was extracted from their message "find pubs with live music" and classified as a soft attribute constraint. The UI shows the backend's interpretation, not the extraction chain.
- **Constraint satisfaction narrative.** `buildAttributeOutcomes` generates text like "3 of 5 mention 'live music' online" but this only appears in the narrative text (if `runReceipt.outcomes.attributes` is populated), not as a structured per-constraint breakdown.

### When Constraints Are Not Visible

If no `constraints_extracted` artefact is returned (which happens on simpler runs), the user sees NO constraint information at all. They have no way to know what the Supervisor interpreted as their requirements.

## 7. Run Status Display

### How Status Is Resolved

`resolveCanonicalStatus()` (`client/src/utils/deliveryStatus.ts`) maps the `delivery_summary.status` string to one of:

| Raw Status | Canonical | User-Facing Label |
|-----------|-----------|-------------------|
| `PASS`, `ACCEPT`, `COMPLETED` | `PASS` | "Complete" (green badge) |
| `PARTIAL` | `PARTIAL` | "Results returned" (blue badge) |
| `STOP` | `STOP` | "Search stopped" (red badge) |
| `FAIL` | `FAIL` | "Verification failed" (red badge) |
| `ACCEPT_WITH_UNVERIFIED` | `ACCEPT_WITH_UNVERIFIED` | "Results returned — not all verified" (amber badge) |
| anything else | `UNAVAILABLE` | "Status unavailable" (grey badge) |

### What Works

- `PASS`, `STOP`, and `FAIL` are meaningfully surfaced with distinct colours and labels.
- `STOP` includes the `stop_reason.message` when present (e.g., "Budget exhausted", "Time predicate cannot be verified").
- `PARTIAL` and `ACCEPT_WITH_UNVERIFIED` have distinct presentations.

### What Does Not Work

- **`COMPLETED` silently maps to `PASS`.** If the backend sends `status: "COMPLETED"`, the user sees "Complete" with a green badge — even if verification was never run and the results are just discovery candidates.
- **`PARTIAL` is ambiguous.** It could mean "found 3 of 5 requested" or "found 5 but only verified 3." The label "Results returned" gives no indication of WHY it's partial.
- **No `ERROR` status.** If the Supervisor crashes, the task row stays in `processing` until a timeout, and the UI shows a spinning state followed by a generic "Something went wrong" — not a meaningful error status.
- **Tower verdict is not surfaced in the main UI.** The `run_receipt.tower_verdict` (e.g., "continue", "revise", "abandon") is available but only shown in `TechnicalDetails`. The user never sees whether the Tower approved or rejected the results.

## 8. Debug/Test Harness

### What Exists

1. **RunDiagnosticPanel** (`chat.tsx`): Dev-only panel toggled by `showDebugPanel`. Shows `conversationId`, message count, `isWaitingForSupervisor`, `inFlightSupervisorRuns`, `google_query_mode`, `lastSentQuery`, and `lastRunFinalVerdict`.

2. **TechnicalDetails** (`RunResultBubble.tsx`): Expandable "Show details" section showing:
   - Tower verdict
   - Constraint table (field, op, value, hardness)
   - Search summary (location, query, counts, stop reason)
   - Policy knobs (search_budget_pages, radius_escalation, learned settings)
   - Learning delta (before/after knob values)
   - Contact counts (emails/phones) with optional receipt comparison

3. **Console logging** in `RunResultBubble`: Enhanced debug log showing `hasVerificationData`, `hasReceiptAttrData`, `hasSemanticData`, `leadVerifications` count, `semanticJudgements` count, `verifiedIds` size, `hasUnverifiedLeads`, `defaultBadgeStatus`, and `renderBranch`.

4. **Activity Panel**: Shows artefact stream with expandable raw JSON views. Can show `VerificationEvidenceView`, `ConstraintsExtractedView`, and `LeadVerificationView` as formatted artefact cards.

5. **Receipt Comparison** (behind `VITE_SHOW_RECEIPT_COMPARISON` env var): Side-by-side UI-derived vs backend receipt contact counts.

### Where It Falls Short

- **No way to replay a run** with the same inputs to compare outputs.
- **No way to inject mock artefacts** to test rendering paths without a real backend run.
- **No assertion framework.** The debug tools DISPLAY data but don't validate correctness. You can see that `hasVerificationData` is false, but nothing alerts when this creates a misleading display.
- **TechnicalDetails is user-hostile.** It's a wall of technical data with no progressive disclosure or "what does this mean" guidance.
- **No way to see what payload was dispatched.** The `lastSentQuery` in the DiagnosticPanel is the user message text, not the full `requestData` that was sent to the Supervisor.
- **No artefact timeline.** You can't see when each artefact arrived or in what order. The Activity Panel shows them as a list but without timestamps.

## 9. Known Weaknesses

### 1. Attributes Silently Dropped on Direct Dispatch
`extractAttributeConstraints()` runs on every message, but on the direct `RUN_SUPERVISOR` path, the extracted attributes are only used for clarification gating — they are never injected into `search_query.attributes`. The Supervisor receives "find pubs in York" without knowing the user asked for "with live music." **The user's constraints are silently lost.**

### 2. "Completed" Masquerades as "Verified"
`COMPLETED` maps to `PASS` → "Complete" with a green badge. A run that merely finished its Google scraping without any verification appears identical to one that the Tower fully verified. The user cannot distinguish "I found stuff" from "I verified stuff."

### 3. Evidence Is Present But Invisible
The rendering code for evidence display (`match_reason`, `evidence[]`, `evidence_source_url`) works correctly — but the data is almost never present on the lead objects in `delivery_summary`. The per-lead evidence lives in `lead_verification` artefacts, but the UI does not cross-reference them into lead row rendering. Most users see results with no evidence at all.

### 4. No Verification ≠ Verified, But Also ≠ Transparent
The recent fix ensures no-verification-data leads show no badge (neutral display), which is correct. But the UI still shows them under "RESULTS (N)" with a green "Complete" status badge — giving a falsely confident overall impression even when no verification occurred.

### 5. Per-Lead Constraint Breakdown Hidden
The richest data — per-lead, per-constraint pass/fail with reasons — exists in `lead_verification` artefacts and is parsed into `leadVerifications`. But it's only shown in `TechnicalDetails` (dev-oriented). The user sees a single badge per lead, not which specific constraints passed or failed.

### 6. Receipt/Narrative Mismatch Risk
`buildAttributeOutcomes` can generate narrative text from either UI-derived counts or `runReceipt.outcomes.attributes`. If the receipt exists, it wins. But if the receipt arrives late (after initial render), the narrative may show one number initially and then update — or not update if the component doesn't re-render.

### 7. Constraint Visibility Is Backend-Dependent
The user only sees constraints if the `constraints_extracted` artefact arrives from the Supervisor. The UI has no fallback to show "here's what I interpreted from your message" from its own extraction logic. If the backend doesn't send this artefact, the user has zero visibility into what requirements were applied.

### 8. `SupervisorTaskData` Interface Is Out of Sync
The TypeScript interface declares `search_query` with only `business_type`, `location`, `requested_count`. The actual runtime object can also contain `attributes`, `opened_proxy`, `proxy_disclaimer`. These fields bypass the type system via `Record<string, any>` casting. Any TypeScript-based validation or autocomplete is useless.

### 9. Error Recovery Exists But Lacks Diagnostic Clarity
The UI does have error/failure states: `still_working`, `artefacts_unavailable`, `run_timeout`, and `run_not_persisted` are all handled in `RunResultBubble` with distinct UI blocks. Retry flows exist via `wyshbone:retry_artefacts` and `wyshbone:retry_last_message` custom events. However, the retry UI gives no diagnostic about WHAT failed — the user sees "Try again" without understanding whether the Supervisor crashed, the verification budget was exhausted, or a network error occurred. Partial results produced before a failure are not surfaced in the error state.

### 10. Direct-Dispatch Intent Persistence Omits Attributes
`saveLastSuccessfulIntent` on the direct `RUN_SUPERVISOR` path (line 1826) persists `business_type`, `location`, and `requested_count` — but NOT `attributes`. This means even if a future fix injects attributes on direct dispatch, followup reuse would still lose them because `lastIntent.attributes` would be undefined.

### 11. CLASSIFY_GATE Path Also Omits Attributes
When `decideChatMode` returns `CLARIFY_FOR_RUN` and the system delegates to Supervisor with `supervisor_preflight`, the `requestData` is constructed without running attribute injection. This creates a third path where user constraints are silently dropped.

### 12. Activity Panel Evidence Is Disconnected
The `VerificationEvidenceView` and `LeadVerificationView` components in the Activity Panel correctly render rich evidence — but they're in a separate panel from the results. A user looking at their results has no link or indication that "you can see the evidence for this in the Activity Panel." The two views are architecturally separated.

---

## Summary

The UI is strongest at: rendering the leads themselves, showing basic status (PASS/STOP/FAIL), and displaying constraints when the backend provides them.

The UI is weakest at: getting its own constraints to the backend (attributes dropped on direct dispatch), showing evidence for verification (data exists but is rarely surfaced), making verification status honest (no-badge neutrality is correct but the overall status badge still implies confidence), and giving users any insight into WHY their results look the way they do.

The fundamental tension: the UI has sophisticated rendering code for verification evidence, constraint status, and per-lead breakdowns — but this code is mostly dormant because the data it needs is either not sent by the UI (attributes), not returned by the backend (evidence on lead objects), or buried in a separate panel the user doesn't know to check (Activity Panel artefacts).
