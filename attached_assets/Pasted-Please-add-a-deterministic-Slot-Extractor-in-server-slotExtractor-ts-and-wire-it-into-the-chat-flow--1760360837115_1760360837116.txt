Please add a deterministic Slot Extractor in server/slotExtractor.ts and wire it into the chat flow BEFORE /api/jobs/create. Keep all existing behavior; just ensure these slots are cleanly populated:

Slots:
{
  query: string,            // e.g., "pubs", "ice cream makers"
  position?: string,        // e.g., "CEO", "director"
  location?: string,        // e.g., "Kendal", "Melbourne", "Texas"
  country?: string,         // canonical name from country CSV ("United Kingdom")
  country_code?: string     // ISO alpha-2 ("GB")
}

Data sources available:
- /server/data/country_codes.csv (Country, Code)
- /server/data/city_hints.json if present

Implementation:
1) Load country CSV on startup into two maps:
   NAME_TO_CODE (normalized country name/alias -> ISO2 uppercase),
   CODE_TO_NAME (ISO2 -> canonical name).
   Aliases: uk/england/scotland/wales/northern ireland -> GB; usa/united states/america -> US, etc.

2) Create function parseFreeText(message: string) -> {raw_query, raw_position, raw_place}
   - Use simple patterns:
     - "... in <place>" captures raw_place
     - position terms: ["ceo","director","owner","manager"] (lowercase match)
     - query: remaining nouns (e.g., "pubs","ice cream makers","dentists")
   - Keep this simple/deterministic (no heavy LLM use here).

3) Create function fillSlots(input) that:
   a) Start with parseFreeText(message).
   b) Try to resolve raw_place as a country:
      - If NAME_TO_CODE hit, set country_code + country = CODE_TO_NAME[country_code]; do NOT ask a question.
   c) If raw_place is not a country:
      - Treat it as location (location = raw_place).
      - If location is a known city in city_hints.json (optional), infer country_code and set country.
   d) If after c) country is still empty AND message mentions a known country elsewhere, set that.
   e) If both country and location are empty, and user gave just one token that is not a country (e.g., "Kendal"), set location=that token, leave country empty and mark needs_clarification.

4) Single clarification rule:
   If country is missing and location is present, generate ONE question:
   "Which country is 'Kendal' in? e.g., United Kingdom (GB)."
   Accept the reply and re-run fillSlots with that answer (no loops).

5) Safety rules:
   - If the user says only a country (e.g., "India"), fill {country="India", country_code="IN"} and set granularity="country", location undefined (OK for Places New using textQuery "X in India").
   - Never ask "is X a country?" — we know from CSV whether it is.

6) Output:
   Return {query, position, location, country, country_code, needs_clarification?: boolean, question?: string}

7) Chat integration:
   - In /api/chat (or the controller), call Slot Extractor on each user message.
   - If needs_clarification === true, stream the one question and wait for user reply.
   - When slots are ready, call /api/jobs/create with:
     { business_type: query, position, country, country_code, granularity (derived by country), region_filter (if location implies state/county/borough), userEmail }

8) Tests (log to console on start):
   - "find pubs in Kendal" -> query="pubs", location="Kendal", country? (unknown) => ask ONE question
   - "find pubs in Kendal, United Kingdom" -> country_code=GB, location=Kendal (no question)
   - "ice cream makers in Vietnam" -> country_code=VN, location undefined (country-level search)
   - "CEO ice cream manufacturer Vietnam" -> position="CEO", query="ice cream manufacturer", country_code=VN
   - "dentists new york" -> location="New York", country_code=US (state-level)
