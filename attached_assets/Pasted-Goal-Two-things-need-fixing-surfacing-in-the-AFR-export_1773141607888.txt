Goal

Two things need fixing / surfacing in the AFR export and QA debugging flow.

1. Fix the small but important loop flaw:
   when the pre-execution constraint gate blocks a run, the system appears to re-enter the mission again and block again, instead of treating the first block as the final benchmark outcome for that run.

2. Surface the full Behaviour evaluation packet and the LLM’s returned judgement in the AFR export, so the export can be externally audited.

Context

In the current AFR export for benchmark runs like:
- Find companies that supply to NHS hospitals in Leeds

the AFR clearly shows:
- mission extraction
- constraints extracted
- capability check
- clarify gate
- pre-execution constraint gate blocked execution

But then the run appears to start another mission cycle and hit the same block again.

That suggests the harness / run reconciliation / benchmark driver is re-triggering evaluation after a terminal benchmark-relevant blocked state, instead of treating that blocked/clarify outcome as the final outcome for the benchmark test.

Also, the AFR export still does not clearly show:
- the Behaviour evaluation packet
- the structured LLM Behaviour result
- what the LLM returned
- why Behaviour passed or failed

We need both fixed.

Do NOT change Tower.
Do NOT loosen benchmark semantics.
Do NOT change the underlying mission planning rules unless necessary to stop the duplicate blocked rerun path for benchmark harness execution.
