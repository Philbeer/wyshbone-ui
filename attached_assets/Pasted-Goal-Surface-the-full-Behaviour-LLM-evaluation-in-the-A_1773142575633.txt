What to build

PART 1 — Determine the true Behaviour source of truth

Audit the current Behaviour pipeline and identify, for each benchmark row:

- whether Behaviour is coming from the new LLM evaluator
- or from fallback legacy logic
- or from some mixed / override path

Persist and expose a field like:

- behaviour_source_of_truth = "llm" | "fallback_legacy" | "unknown"

Also expose:
- behaviour_eval_mode
- behaviour_fallback_used (true/false)
- fallback_reason if fallback was used

This is critical because I need to know whether the UI Behaviour result is really being dictated by the LLM call or not.

PART 2 — Persist the exact Behaviour LLM input packet

When the Behaviour LLM evaluator runs, persist the normalized input packet it receives.

I want this packet available to AFR export in a readable form.

Required fields to persist and export if available:

- benchmark_test_id
- original_query
- expected_outcome_text
- expected_behaviour_text
- actual_run_state
- clarified
- clarify_question
- clarify_answer
- tower_result
- delivered_count
- delivered_entities
- evidence_summary
- behaviour_observed_summary

This is the packet that was actually sent or prepared for the LLM judge.
Do not dump the full AFR raw run.
Use the compact normalized packet actually used by the evaluator.

Persist it as something like:
- behaviour_eval_packet

PART 3 — Persist the exact Behaviour LLM response

When the Behaviour LLM returns, persist:

- parsed structured response
- raw model response text if available
- parse success / failure state

Required parsed fields:

- behaviour_result
- behaviour_reason
- expected_outcome_check
- observed_outcome_check
- key_failure_type
- confidence

Persist as:
- behaviour_eval_response
- behaviour_eval_response_raw
- behaviour_eval_parse_ok
