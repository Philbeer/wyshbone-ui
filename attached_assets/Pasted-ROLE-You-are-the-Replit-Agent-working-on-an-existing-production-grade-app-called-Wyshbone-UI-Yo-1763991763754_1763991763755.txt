SECTION 3 – DATA MODEL
============================================================

Follow existing DB conventions. If a Supabase/Postgres layer exists, extend it accordingly.

There are two conceptual layers:

- **Core CRM entities** – generic.
- **Brewery-specific entities** – only used for `industry_vertical = "brewery"`.

You may use the naming below or adapt to existing patterns, but keep the relationships.

### 3.1 Core CRM entities

1) `CrmCustomer` (or extend existing "Customer" table)
   - id (uuid)
   - workspace_id (fk -> workspace/account)
   - name (text)           // business name
   - primary_contact_name (text, nullable)
   - email (text, nullable)
   - phone (text, nullable)
   - address_line1 (text, nullable)
   - address_line2 (text, nullable)
   - city (text, nullable)
   - postcode (text, nullable)
   - country (text, default "United Kingdom")
   - notes (text, nullable)
   - created_at, updated_at

2) `CrmOrder`
   - id (uuid)
   - workspace_id
   - customer_id (fk -> CrmCustomer)
   - order_number (text)             // simple incremental per workspace
   - order_date (date)
   - status (enum: "draft", "confirmed", "dispatched", "delivered", "cancelled")
   - delivery_date (date, nullable)
   - delivery_run_id (fk -> DeliveryRun, nullable)
   - currency (text, default "GBP")
   - total_amount (numeric, nullable – can be calculated from lines)
   - notes (text, nullable)
   - created_at, updated_at

3) `CrmOrderLine`
   - id (uuid)
   - order_id (fk -> CrmOrder)
   - generic_item_name (text)    // generic label, e.g. "Consultation", "Product X"
   - generic_item_code (text, nullable)
   - quantity_units (numeric)    // quantity (units, hours, etc.)
   - unit_price (numeric)
   - line_total (numeric)
   - vertical_type (enum/string, nullable) // e.g. "brewery" when mapped to a Product
   - vertical_ref_id (uuid, nullable)      // fk into vertical-specific table (e.g. BrewProduct)
   - created_at, updated_at

4) `CrmDeliveryRun`
   - id (uuid)
   - workspace_id
   - name (text)                 // e.g. "London Central Route", or "Delivery Run – 2025-11-24"
   - driver_name (text, nullable)
   - vehicle (text, nullable)
   - scheduled_date (date)
   - status (enum: "planned", "in_progress", "completed", "cancelled")
   - notes (text, nullable)
   - created_at, updated_at

5) Generic `CrmSettings`
   - id (uuid)
   - workspace_id
   - industry_vertical (enum/string as above)
   - default_country (text, default "United Kingdom")
   - created_at, updated_at

### 3.2 Brewery-specific entities (for vertical = "brewery")

1) `BrewProduct`
   - id (uuid)
   - workspace_id
   - name (text)
   - style (text, nullable)
   - sku (text, nullable, unique per workspace)
   - abv (numeric)
   - default_package_type (enum: "cask", "keg", "can", "bottle")
   - default_package_size_litres (numeric)   // e.g. 40.9 for firkin
   - duty_band (enum/string, e.g. "beer_standard", "beer_small_producer")
   - is_active (boolean, default true)
   - created_at, updated_at

2) `BrewBatch`
   - id (uuid)
   - workspace_id
   - product_id (fk -> BrewProduct)
   - batch_code (text)   // e.g. LS2401
   - brew_date (date)
   - status (enum: "planned", "in_progress", "fermenting", "packaging", "packaged", "cancelled")
   - planned_volume_litres (numeric)
   - actual_volume_litres (numeric, nullable)
   - notes (text, nullable)
   - created_at, updated_at

3) `BrewInventoryItem`
   - id (uuid)
   - workspace_id
   - product_id
   - batch_id (nullable)
   - package_type (enum: "cask", "keg", "can", "bottle")
   - package_size_litres (numeric)
   - quantity_units (integer)
   - location (text)           // e.g. "Cold Store", "Main Warehouse"
   - created_at, updated_at

4) `BrewContainer`
   - id (uuid)
   - workspace_id
   - container_code (text)     // physical identifier
   - container_type (enum: "cask", "keg")
   - volume_litres (numeric)
   - status (enum: "at_brewery", "with_customer", "lost", "retired")
   - last_customer_id (fk -> CrmCustomer, nullable)
   - last_outbound_date (date, nullable)
   - last_return_date (date, nullable)
   - notes (text, nullable)
   - created_at, updated_at

5) `BrewDutyReport`
   - id (uuid)
   - workspace_id
   - period_start (date)
   - period_end (date)
   - total_litres (numeric)
   - total_duty_amount (numeric)
   - breakdown_json (jsonb)   // per product/abv band
   - created_at, updated_at

6) `BrewSettings`
   - id (uuid)
   - workspace_id
   - default_warehouse_location (text, nullable)
   - default_duty_rate_per_litre (numeric, nullable)
   - created_at, updated_at

NOTE: For other verticals (e.g. animal physio), we will later add their own tables like `PhysioService`, `PhysioAppointment`, etc. The core CRM entities stay shared.

SECTION 4 – BACKEND API & ACTION ENDPOINTS
============================================================

Create REST or RPC-style endpoints under a clear namespace, following existing API conventions.

### 4.1 Core CRM endpoints

- `/api/crm/customers`       (GET list, POST create)
- `/api/crm/customers/:id`   (GET, PUT/PATCH, DELETE/soft-delete)
- `/api/crm/orders`
- `/api/crm/orders/:id`
- `/api/crm/delivery-runs`
- `/api/crm/delivery-runs/:id`
- `/api/crm/settings` (for CrmSettings)

All list GETs:
- Filter by `workspace_id` from authenticated user context.
- Support basic filtering (search, date range, status) where obvious.

### 4.2 Brewery vertical endpoints

- `/api/brewcrm/products`
- `/api/brewcrm/products/:id`
- `/api/brewcrm/batches`
- `/api/brewcrm/batches/:id`
- `/api/brewcrm/inventory`
- `/api/brewcrm/containers`
- `/api/brewcrm/containers/:id`
- `/api/brewcrm/duty-reports`
- `/api/brewcrm/duty-reports/:id`
- `/api/brewcrm/settings`

Only allow access if `industry_vertical === "brewery"` for the current workspace.

### 4.3 Agent-style action endpoints

To make the system agent-ready, implement action endpoints that perform higher-level operations:

1) `POST /api/crm/actions/create_customer`
   - Body:
     - name, primary_contact_name, email, phone, address fields
   - Returns: created CrmCustomer.

2) `POST /api/brewcrm/actions/create_product`
   - Body:
     - name, style, sku, abv, default_package_type, default_package_size_litres, duty_band
   - Returns: created BrewProduct.

3) `POST /api/brewcrm/actions/create_batch`
   - Body:
     - product_id
     - batch_code
     - brew_date
     - planned_volume_litres
   - Returns: created BrewBatch.

4) `POST /api/crm/actions/create_order`
   - Body:
     - customer_id
     - order_date
     - optional delivery_date
     - lines: [
         { generic_item_name, generic_item_code, quantity_units, unit_price, vertical_type, vertical_ref_id }
       ]
   - For brewery vertical, vertical_type = "brewery" and vertical_ref_id = BrewProduct.id.
   - Returns: created CrmOrder with lines.

5) `POST /api/crm/actions/assign_order_to_delivery_run`
   - Body:
     - order_id
     - delivery_run_id
   - Returns: updated CrmOrder.

6) `POST /api/crm/actions/mark_delivery_complete`
   - Body:
     - delivery_run_id
     - optional actual_completion_date (default = today)
   - Side effects:
     - Set DeliveryRun.status = "completed".
     - Set all associated CrmOrders to status "delivered".
   - Returns: updated DeliveryRun.

7) `POST /api/brewcrm/actions/generate_duty_report`
   - Body:
     - period_start
     - period_end
   - Behaviour:
     - Aggregate packaged volumes by ABV/duty band from BrewBatch/BrewInventory/BrewProduct as appropriate.
     - Calculate total duty_amount using BrewSettings.default_duty_rate_per_litre (simple model).
     - Create BrewDutyReport row and return it.

All action endpoints:
- Validate workspace ownership from auth context.
- Return structured JSON with `success`, `data`, and optionally `errors`.

SECTION 7 – UX PRINCIPLES
Simplicity first

Target users are small teams who might otherwise use spreadsheets.

Fewer, clearer forms beat many advanced options.

Consistent terminology

For generic verticals use neutral labels: Customer, Order, Delivery Run.

For brewery vertical:

Use "Beer" or "Product" consistently in UI copy, but keep DB names as in section 3.

Use "Batch" and "Delivery Run" in titles and buttons.

Easy “happy path”

Brewery happy path:

Create Product → Plan Batch → Mark Packaged → Create Order → Assign to Run → Mark Delivered → Generate Duty.

Generic happy path:

Create Customer → Create Order → Assign to Delivery Run → Mark Delivered.

Feedback & errors

Forms must validate required fields and show clear errors.

Successful actions should show toasts/snackbars.
