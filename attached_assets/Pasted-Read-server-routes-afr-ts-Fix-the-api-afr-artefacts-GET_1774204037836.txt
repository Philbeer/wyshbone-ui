Read server/routes/afr.ts. Fix the /api/afr/artefacts GET endpoint so it finds supervisor artefacts BEFORE the supervisor_run_id bridge is set up.

ROOT CAUSE: When the UI calls /api/afr/artefacts?client_request_id=X:
1. It resolves X → agent_run.id (e.g. "abc")
2. Queries artefacts WHERE run_id = 'abc'
3. Tries supervisor_run_id fallback — but that column is NULL for ~8-12 seconds
4. Returns empty — the intent_narrative artefact is under the supervisor's own run_id

The supervisor logs agent_activities with the same client_request_id but with run_id = its OWN supervisor run_id. We can extract that from activities.

FIX: After the initial artefact query, if we don't have intent_narrative yet, find ALL unique run_ids from agent_activities for this client_request_id, and query artefacts for any we haven't checked.
