This is still broken. Live Activity shows “Completed” immediately for a brand new request, before any real events for that request exist.

This is not a UI timing issue anymore. It means one of these is true:
1) activeClientRequestId is not actually changing per user request (stale request id reused)
2) /api/afr/stream is returning is_terminal=true by default or returning terminal state from the previous run
3) Client is caching terminal state and not resetting it on new request
4) The stream query is not properly filtered by client_request_id (it is returning the most recent terminal run, not the current run)

We need a hard rule:
A run can NEVER be “Completed” if it has 0 events for the current activeClientRequestId.
