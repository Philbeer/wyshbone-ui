SUPERVISOR REPO TASK: reduce brittle semantic interpretation and trust canonical intent instead.

Goal
Simplify Supervisor so it stops re-parsing natural language and pre-deciding semantic meaning that should belong to the canonical intent extractor or Tower.

Architecture target
- Canonical intent extractor owns language understanding
- Supervisor owns tool selection, execution flow, retries, stopping, and safety rails
- Tower will own semantic judgement of whether evidence satisfies the user goal

Implement a safe first reduction pass.

What to change
1. Replace downstream raw-text semantic re-interpretation with canonical intent fields wherever possible.
Use canonical intent output instead of rescanning user text for:
- subjective constraints
- attribute classification
- time predicate detection
- relationship predicate detection
- hardness assignment
- follow-up semantic scope comparison

2. Keep only non-semantic deterministic rails in Supervisor:
- gibberish / empty input detection
- coordinate / bounding box checks
- name matching
- location sanitisation
- vertical safety rails
- honesty guardrails
- simple command detection like "go ahead"

3. For each brittle semantic subsystem currently duplicating LLM work:
- prefer canonical intent output if available
- only fall back to old regex logic if canonical intent is missing/invalid
- add a clear source marker in logs/probes:
  - semantic_source: canonical
  - semantic_source: fallback_regex

4. Do NOT yet move Tower judgement in this task.
This task is only about stopping Supervisor from reinterpreting language when canonical intent already exists.

Priority targets from the report
- clarify-gate semantic regex routing
- subjective term detection
- attribute phrase classification
- time predicate detection
- relationship predicate detection
- serve/serves disambiguation
- hardness assignment
- follow-up contradiction / scope-change semantic parsing

Important
- minimal production-grade change
- keep fallback safety
- do not break working runs
- do not redesign the whole repo
- do not remove deterministic execution logic

Success result
For a query like:
"Find cafes in Manchester that serve vegan food"

Supervisor should trust the canonical intent:
- cafes
- Manchester
- attribute constraint: vegan food

and should not reclassify "serve" as a relationship predicate via brittle downstream rules.
